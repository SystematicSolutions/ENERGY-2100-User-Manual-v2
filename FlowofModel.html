import ModuleName - loads a specific module or package into the current environment, allowing to use its functionalities.


Purpose: It is used to bring an external module and its names into the scope of the current module, allowing you to use its functionalities without naming conflicts.

  • Scope:
    • import ModuleName only brings the module's name (identifier) itself into the current scope. To access its contents, you must use qualified names, like ModuleName.function_name().
    • In contrast, using ModuleName (which is often used alongside or instead of import for convenience) brings all the module's exported names directly into the current scope, allowing you to use function_name() directly.
  • Usage Context: You use import (or using) to use external libraries or submodules defined elsewhere, which are often precompiled and managed by Julia's package manager.


The code shares the scope of the calling module, so all variables an dfunctions defined in the included file are available in the main file's scope by default.Include:  It is as if the code from the included file was written directly at the location of the include statement.

Usage


  • Running Julia: You can run Julia by typing julia in your terminal or command prompt if the executable's directory is added to your system's PATH environment variable.
  • IDE Configuration: When using an Integrated Development Environment (IDE) like VS Code, you often need to manually set the julia.executablePath setting to point to this file so the extension can function correctly.
  • Checking the Path: From within the Julia REPL (interactive command line), you can find the path to the currently running executable using the built-in function Sys.which("julia") or Sys.BINDIR. 



When running a script using julia, you can pass additional arguments to your script:


$ julia script.jl arg1 arg2...

\src\EnergyModel.jl

  • Core Code: It is intended for the code that defines functions, types, and modules which provide the main functionality of your project.
  • Main Entry Point: In a package named YourPackageName, the src folder will typically contain a file named YourPackageName.jl (or a similar name if not using PkgTemplates.jl). This file serves as the main entry point and module definition for your package, which then uses include to incorporate other source files within the src folder as necessary.
  • Separation of Concerns: Files in src should ideally not produce output (like plots or data files) when included; they should only define functionality. Any scripts that generate output or run specific pipelines should generally be placed in a separate scripts/ or bin/ directory.
  • Standard Structure: While not strictly mandatory for very small scripts, using a src folder, along with a Project.toml file, is a standard practice and is highly recommended for proper package development, dependency management, and sharing. Other standard folders include test/ for testing and docs/ for documentation.
